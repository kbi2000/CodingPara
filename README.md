# CodingPara Technologies website

Production-oriented Next.js App Router website for CodingPara Technologies. Content is held in typed data modules, pages are server-rendered or statically generated by default, and interactive JavaScript is limited to navigation, filtering, consent and forms.

## Local development

1. Install Node.js 22 LTS.
2. Copy `.env.example` to `.env.local` and configure the required values.
3. Run `npm install`, then `npm run dev`.
4. Before release run `npm run lint`, `npm run typecheck`, `npm run format:check` and `npm run build`.

## Content editing

Business details, services, FAQs, articles, projects and genuine job openings live in `src/data/site.ts`. Keep statistics `null` until verified. Add testimonials only after receiving approval. Blog content currently uses typed entries and route rendering; the module boundary can be replaced by MDX, Sanity, Strapi, Contentful, Directus or headless WordPress without redesigning page components.

An open job may use `status: "open"`; closed jobs must use `closed` or be removed. Add JobPosting JSON-LD only alongside a complete, visible open-job description.

## Lead delivery

`POST /api/leads` validates all input server-side, rejects honeypots and implausibly fast submissions, and applies a basic per-instance rate limit. Valid contact and SEO-audit submissions are sent to a free Google Apps Script web app. The script appends the lead to Google Sheets and emails both configured recipients.

### Google Sheets setup

1. Create a Google Sheet and rename one tab to `Leads`.
2. Paste this header row into row 1: `kind`, `name`, `email`, `phone`, `service`, `budget`, `timeline`, `message`, `consent`, `startedAt`, `receivedAt`, `ipAddress`, `userAgent`.
3. Copy the spreadsheet ID from the URL between `/d/` and `/edit`.
4. Open **Extensions → Apps Script** and replace the editor contents with [`integrations/google-apps-script/Code.gs`](integrations/google-apps-script/Code.gs).
5. In Apps Script, open **Project Settings → Script properties** and add:
   - `SPREADSHEET_ID`: the ID copied in step 3
   - `SHEET_NAME`: `Leads`
   - `LEADS_TO_EMAIL`: `info@codingpara.com,helloworldkabir@gmail.com`
   - `WEBHOOK_SECRET`: a long random secret you create
6. Select **Deploy → New deployment → Web app**. Execute as yourself and allow access to anyone who can call the web app. Authorize Sheets and email access, deploy, and copy the URL ending in `/exec`.
7. Configure the website with the same URL and secret:

```env
GOOGLE_APPS_SCRIPT_URL=https://script.google.com/macros/s/.../exec
GOOGLE_APPS_SCRIPT_SECRET=the-same-long-random-secret
```

Restart the development server after changing `.env.local`. For production, add the same variables to the hosting provider. Never expose the secret as a `NEXT_PUBLIC_*` variable or commit `.env.local`.

The script uses a lock to prevent simultaneous submissions from overwriting rows and neutralises values that could be interpreted as spreadsheet formulas. For multi-instance production, replace the in-memory rate limiter with a shared service such as Upstash Redis or Cloudflare.

## Analytics and consent

Conversion interactions dispatch `codingpara:analytics` browser events. Connect the listener to GTM/GA4, Clarity or Meta only after the user selects analytics consent. IDs are represented in `.env.example`; no trackers load by default.

## Deployment

- Vercel: import the repository, add environment values, build with `npm run build`.
- Docker/VPS: set `output: "standalone"` in `next.config.ts`, build with `docker build -t codingpara .`, then run behind TLS using the Nginx example.
- AWS: deploy the container to ECS/App Runner or use an adapter suitable for CloudFront and Lambda.
- Cloudflare: use the current OpenNext Cloudflare adapter and test route handlers in its runtime before release.

Health checks use `/api/health`. Keep the previous deployment image and environment snapshot. Roll back by routing traffic to the last healthy deployment; restore lead data independently from its managed database backup.

## SEO migration checklist

- Export the final WordPress URL, title, description, canonical, status and backlink inventory before cutover.
- Review `next.config.ts` redirects against the full crawl; add every changed public URL as a permanent redirect.
- Preserve valuable content and media paths where licensed and practical.
- Test one-hop redirects, canonical URLs, robots.txt, XML sitemap, structured data and every indexable response code in staging.
- Keep staging blocked at the platform/auth layer, not via production metadata copied to launch.
- Verify both domain properties in Search Console, submit `/sitemap.xml`, inspect priority URLs and monitor coverage, Core Web Vitals and 404s after launch.
- Do not remove the old deployment until DNS, forms, analytics and rollback have been tested.

## Performance and release checklist

- Test 320, 375, 425, 768, 1024, 1280, 1440 and 1920 px widths for overflow and keyboard navigation.
- Test menu, dropdown, accordions, portfolio filters, consent, forms and error states.
- Run Lighthouse mobile against production-like hosting; confirm LCP, CLS and INP targets.
- Verify each page has one H1, a unique title/description, canonical, visible breadcrumbs where appropriate and valid internal links.
- Confirm form delivery, rate limiting, CSP reports, external links, 404, redirects and health check.
- Run dependency audit and update intentionally before release.
